Modular in 3 Layers

Arquitetura modular com menos acoplamento e mais clareza.

M3L é um padrão arquitetural para backends que organiza a aplicação por módulos de negócio e separa responsabilidades em três camadas: Http, Domain e Infrastructure.

O que é M3L

M3L significa Modular in 3 Layers. O padrão foi pensado para manter o código legível, previsível e sustentável, evitando depósitos centrais de classes, acoplamento desnecessário e responsabilidades espalhadas sem critério.

Módulo primeiro

O sistema é organizado por contexto de negócio, não por pastas globais genéricas.

Camada depois

Cada módulo separa entrada HTTP, orquestração de caso de uso e persistência/leitura.

Responsabilidade clara

Controller recebe. Service orquestra. Business decide. Model persiste. Query lê.

Princípios centrais

O objetivo não é parecer sofisticado. É ser difícil de bagunçar. Arquitetura boa é a que aguenta o time, o prazo e a vida real.

01

Baixo acoplamento

Os módulos se organizam com clareza, reduzindo dependências desnecessárias e cruzamentos caóticos.

02

Coesão estrutural

Cada camada tem papel definido. Quando tudo pode tudo, o projeto começa a pedir socorro em silêncio.

03

Evolução sustentável

O padrão foi feito para crescer sem transformar o código em um labirinto com nomes bonitos.

04

Pragmatismo

M3L não é exercício acadêmico. É disciplina prática para sistemas que precisam continuar compreensíveis.

Manifesto resumido

M3L não existe para deixar o projeto bonito. Existe para deixar o projeto entendível, revisável e evolutivo. A regra é simples: cada arquivo deve nascer na camada certa e usar o framework na dose certa.

Próximo passo

Comece pelo repositório da stack que você usa. Leia o README, veja a estrutura proposta e adapte com disciplina. Arquitetura boa não é a que impressiona no primeiro dia; é a que continua fazendo sentido no dia 200.
